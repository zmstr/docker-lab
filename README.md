# Home Lab — Docker Stack

A single Docker Compose stack for a self-hosted home lab on an Intel NUC running Ubuntu.

**Services**

| Service | URL | Purpose |
|---|---|---|
| Traefik | https://traefik.lan | Reverse proxy & TLS termination |
| WireGuard Easy | https://wg.lan | VPN server management UI |
| Vaultwarden | https://pass.lan | Self-hosted password manager |
| Home Assistant | https://hass.lan | Home automation platform |
| Portainer | https://portainer.lan | Docker management UI |

## Architecture

```
Internet / LAN
       │
  ┌────▼─────┐
  │  Traefik │  :80 (→303 HTTPS)  :443 (TLS termination)
  └────┬─────┘
       │  Docker internal network "proxy" (172.20.0.0/24)
       ├──────────────────────────────────────────────┐
       │                                              │
  ┌────▼──────┐  ┌────────────┐  ┌────────────┐  ┌───▼────────┐
  │ wg-easy   │  │ vaultwarden│  │homeassistant│  │ portainer  │
  │ :51821    │  │ :80        │  │ :8123       │  │ :9000      │
  └─────┬─────┘  └────────────┘  └────────────┘  └────────────┘
        │ UDP :51820 (direct — bypasses Traefik, VPN traffic only)
```

- Traefik is the **only** container with ports bound on all interfaces.
- All web UIs are accessible exclusively through Traefik over HTTPS.  
- WireGuard VPN UDP traffic bypasses Traefik (Traefik cannot proxy UDP).
- TLS is terminated at Traefik using certificates generated by **mkcert**.

### Certificate Strategy — mkcert

Since the stack is not exposed to the internet, Let's Encrypt cannot validate domain ownership. The recommended approach is **mkcert**, a zero-config local CA tool:

- Creates a **local Certificate Authority (CA)** on the NUC.
- Issues certificates that browsers and OS trust **after you import the CA once per device**.
- No browser warnings, no annual renewals, one cert covers all `.lan` domains.
- Completely offline — no dependency on external services.

> **Alternatives explored:**  
> - *Traefik built-in self-signed cert* — works out of the box but always shows browser warnings; not suitable for Vaultwarden browser extension.  
> - *step-ca container* — full PKI infrastructure, excellent for larger setups but overkill for a single-host home lab.  
> - *Web generators (SSL For Free, ZeroSSL)* — require public domain ownership validation; incompatible with `.lan` domains.  
> 
> **mkcert wins** for home labs: simple, trustable, zero-infrastructure.

## Step 1 — Install Docker

### 1.1 Remove old packages

```bash
sudo apt remove -y docker docker-engine docker.io containerd runc
```

### 1.2 Install Docker Engine via the official repository

```bash
sudo apt update
sudo apt install -y ca-certificates curl gnupg lsb-release

sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

### 1.3 Verify the installation

```bash
sudo docker run --rm hello-world
docker compose version   # should print v2.x or later
```

### 1.4 Add your user to the docker group

> **Security note:** members of the `docker` group have effective root access to the host. Ensure only trusted accounts are added.

```bash
sudo usermod -aG docker tom
newgrp docker   # apply group change in the current shell
```

### 1.5 Enable Docker to start on boot

```bash
sudo systemctl enable docker
sudo systemctl enable containerd
```

## Step 2 — Harden Docker

### 2.1 Configure the Docker daemon

Create or edit `/etc/docker/daemon.json`:

```bash
sudo nano /etc/docker/daemon.json
```

Paste the following (adjust `log-opts` to your preference):

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "icc": false,
  "no-new-privileges": true,
  "userland-proxy": false
}
```

| Option | Effect |
|---|---|
| `log-driver / log-opts` | Prevent logs from filling the disk |
| `icc: false` | Disable direct container-to-container communication except via explicit networks |
| `no-new-privileges: true` | Global default — containers cannot escalate privileges via setuid |
| `userland-proxy: false` | Use kernel NAT instead of a Docker-managed proxy daemon |

Restart Docker to apply:

```bash
sudo systemctl restart docker
```

### 2.2 Keep the host OS updated

```bash
sudo apt update && sudo apt upgrade -y
# Optional — enable automatic security updates:
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
```

### 2.3 Set up a firewall (UFW)

```bash
sudo apt install -y ufw

# Default: deny all incoming, allow all outgoing
sudo ufw default deny incoming
sudo ufw default allow outgoing

# SSH — adjust the port if you changed it
sudo ufw allow 22/tcp

# Traefik
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# WireGuard VPN (UDP only)
sudo ufw allow 51820/udp

# Enable the firewall
sudo ufw enable
sudo ufw status verbose
```

> **Note:** Docker bypasses UFW for container port bindings by default. The compose file uses `127.0.0.1` binding for container ports that should not be directly reachable, and UFW rules cover what's left.

### 2.4 Docker socket security

The Docker socket (`/var/run/docker.sock`) grants root-level access to the host. In this stack:

- **Traefik** mounts it **read-only** (`:ro`) — sufficient for service discovery.
- **Portainer** mounts it **read-only** (`:ro`) — Portainer uses the API, which is readable via the socket even in read-only mode for management operations.

For an extra layer of security, consider running a **Docker socket proxy** (e.g., `tecnativa/docker-socket-proxy`) to whitelist only the API endpoints each container needs. This is recommended if you want to harden further in the future.

### 2.5 Limit container resources (optional but recommended)

Add resource limits to each service in `docker-compose.yml` to prevent a misbehaving container from starving the host:

```yaml
# Example — add under any service:
deploy:
  resources:
    limits:
      memory: 512m
      cpus: "0.5"
```

## Step 3 — Generate TLS Certificates

### 3.1 Install mkcert

```bash
sudo apt update && sudo apt install -y libnss3-tools

# Download the latest release (check https://github.com/FiloSottile/mkcert/releases)
curl -Lo mkcert https://github.com/FiloSottile/mkcert/releases/latest/download/mkcert-v1.4.4-linux-amd64
chmod +x mkcert
sudo mv mkcert /usr/local/bin/mkcert

mkcert --version
```

### 3.2 Install the local CA

```bash
mkcert -install
```

This creates a local Certificate Authority and registers it with the system trust store (NSS, for Chrome/Chromium). Firefox uses its own trust store — see section 3.4.

### 3.3 Generate the certificate

```bash
mkdir -p /home/tom/traefik/certs
cd /home/tom/traefik/certs

mkcert \
  -cert-file local-cert.pem \
  -key-file  local-key.pem \
  wg.lan pass.lan hass.lan portainer.lan traefik.lan
```

This produces two files:
- `local-cert.pem` — the certificate (valid for all five domains)
- `local-key.pem` — the private key

### 3.4 Trust the CA on other devices

The mkcert CA root certificate must be imported on every device that will access the services (laptop, phone, tablet). Without this step: connection will work but the browser will show a security warning.

**Find the CA file:**
```bash
mkcert -CAROOT
# Prints the path, e.g. /home/tom/.local/share/mkcert
# The CA file you need: <path>/rootCA.pem
```

Copy `rootCA.pem` to each device and import it:

| Platform | How to import |
|---|---|
| **Linux (Chrome/Chromium)** | Already trusted after `mkcert -install` on the NUC |
| **Linux (Firefox)** | Settings → Privacy & Security → Certificates → View Certificates → Authorities → Import |
| **Windows** | Double-click `rootCA.pem` → Install Certificate → Trusted Root Certification Authorities |
| **macOS** | Double-click `rootCA.pem` → Keychain Access → mark as "Always Trust" |
| **iOS** | AirDrop or email `rootCA.pem` → Settings → Profile Downloaded → Install; then Settings → General → About → Certificate Trust Settings → enable |
| **Android** | Settings → Security → Encryption & Credentials → Install a certificate → CA Certificate |

## Step 4 — Configure DNS

The `.lan` domains need to resolve to the NUC's LAN IP address. Find the NUC's IP:

```bash
ip -4 addr show | grep inet
# Example output: 192.168.1.100
```

### Option A — Edit `/etc/hosts` (per device, no additional software)

Add these lines on **each** device that needs access (Linux/macOS: `/etc/hosts`; Windows: `C:\Windows\System32\drivers\etc\hosts`):

```
192.168.1.100  traefik.lan
192.168.1.100  wg.lan
192.168.1.100  pass.lan
192.168.1.100  hass.lan
192.168.1.100  portainer.lan
```

Native Android and iOS cannot edit the hosts file — you need Option B for mobile access.

### Option B — Local DNS server (recommended for whole-home access)

Run a local DNS server (e.g., **Pi-hole**, **AdGuard Home**, or **dnsmasq**) and add the above mappings as local DNS records. Point your router's DHCP to hand out the NUC's IP as the DNS server. All devices on the LAN resolve the domains automatically — including mobile devices.

## Step 5 — Create the File Structure

Run on the NUC as user `tom`:

```bash
# Traefik
mkdir -p /home/tom/traefik/{certs,config,logs}

# Application data
mkdir -p /home/tom/wg-easy
mkdir -p /home/tom/vaultwarden
mkdir -p /home/tom/homeassistant
mkdir -p /home/tom/portainer

# Set ownership
sudo chown -R tom:tom /home/tom/
```

Copy the configuration files from this repository to the NUC:

```bash
# From the machine where you clone this repo:
scp traefik/traefik.yml       tom@192.168.1.100:/home/tom/traefik/traefik.yml
scp traefik/config/dynamic.yml tom@192.168.1.100:/home/tom/traefik/config/dynamic.yml
```

Or clone the repository directly on the NUC:

```bash
git clone https://github.com/<your-user>/docker-tom.git ~/homelab
cp ~/homelab/traefik/traefik.yml          /home/tom/traefik/traefik.yml
cp ~/homelab/traefik/config/dynamic.yml   /home/tom/traefik/config/dynamic.yml
```

## Step 6 — Configure Environment Variables

```bash
cd ~/homelab   # or wherever you store docker-compose.yml
cp .env.example .env
nano .env
```

Fill in the values:

| Variable | Description | How to generate |
|---|---|---|
| `TIMEZONE` | Your timezone, e.g. `Europe/Brussels` | [TZ list](https://en.wikipedia.org/wiki/List_of_tz_database_time_zones) |
| `WG_HOST` | Public IP or DDNS hostname of the NUC | Check your router or a DDNS service |
| `WG_PASSWORD_HASH` | bcrypt hash of the wg-easy UI password | See below |
| `VAULTWARDEN_ADMIN_TOKEN` | Admin panel secret token | `openssl rand -base64 48` |

**Generate WireGuard password hash:**

```bash
docker run --rm -it ghcr.io/wg-easy/wg-easy wgpw YOUR_PASSWORD
```

Copy the entire `$2b$...` string and paste it as the value of `WG_PASSWORD_HASH`.

## Step 7 — Configure the Traefik Dashboard Password

The dashboard password is stored as an Apache `htpasswd` hash in `traefik/config/dynamic.yml` on the NUC. This avoids any shell variable escaping issues.

**Install htpasswd:**

```bash
sudo apt install -y apache2-utils
```

**Generate a hash:**

```bash
htpasswd -nb admin YOUR_DASHBOARD_PASSWORD
# Output example: admin:$apr1$ruca84Hq$mbjdMZBAG.KWn7vfN/SNK/
```

**Edit the dynamic config on the NUC:**

```bash
nano /home/tom/traefik/config/dynamic.yml
```

Replace the placeholder line under `dashboard-auth → users`:

```yaml
        users:
          - "admin:$apr1$ruca84Hq$mbjdMZBAG.KWn7vfN/SNK/"
```

Traefik will reload the file automatically — no restart needed.

## Step 9 — First-Time Service Configuration

### Traefik Dashboard

Open https://traefik.lan and log in with the credentials from Step 7. Verify that all five routers show a green status.

### WireGuard Easy

1. Open https://wg.lan and log in with the password you hashed in Step 6.
2. Create your first VPN client: click the **+** button, give it a name, and download the `.conf` or QR code.
3. For external (internet) access, forward **UDP port 51820** on your home router to the NUC's LAN IP.

### Vaultwarden

1. Open https://pass.lan and create your **admin account first** (registration is open on first use). Set `SIGNUPS_ALLOWED=false` in `.env` before sharing access.
2. Access the admin panel at https://pass.lan/admin using the `VAULTWARDEN_ADMIN_TOKEN` from your `.env`.
3. Install the [Bitwarden browser extension](https://bitwarden.com/download/) and point it to `https://pass.lan` as the self-hosted server URL.

### Home Assistant

Home Assistant must be told to trust the Traefik reverse proxy, otherwise it will reject requests.

After the container has started for the first time:

```bash
nano /home/tom/homeassistant/configuration.yaml
```

Add the following block (append if the `http:` key already exists):

```yaml
http:
  use_x_forwarded_for: true
  trusted_proxies:
    - 172.20.0.0/24   # proxy Docker network (fixed in docker-compose.yml)
```

Restart the container:

```bash
docker compose restart homeassistant
```

Open https://hass.lan and complete the onboarding wizard.

> **USB devices (Zigbee / Z-Wave):** If you attach a USB stick, uncomment `privileged: true` in `docker-compose.yml`, or add a `devices:` entry pointing to `/dev/ttyUSB0` (or the correct device path). Reconnect the container with `docker compose up -d --force-recreate homeassistant`.

### Portainer

1. Open https://portainer.lan.
2. On first visit you have 5 minutes to create the admin account before Portainer locks itself.
3. Select **"Get started"** to manage the local Docker environment.

## Updating Containers

### Method 1 — Command line (recommended)

```bash
cd ~/homelab

# Pull the latest image versions
docker compose pull

# Recreate only containers whose image changed
docker compose up -d --remove-orphans

# Remove old unused images to free disk space
docker image prune -f
```

### Method 2 — Portainer

1. Open https://portainer.lan.
2. Go to **Stacks** → select the `homelab` stack.
3. Pull the latest images with the **"Pull and redeploy"** button.

### Checking what update is available

```bash
# Show current image digests
docker compose images

# Or use watchtower for a one-off check (does not apply updates automatically)
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  containrrr/watchtower --run-once --monitor-only
```

## Backup & Restore

All persistent data lives in `/home/tom/<container-name>/`. A consistent backup strategy:

### Simple tarball backup

```bash
# Stop the stack first for consistency (optional but recommended for databases)
docker compose stop

# Create a dated backup
BACKUP_DATE=$(date +%Y-%m-%d)
sudo tar -czf ~/backups/homelab-${BACKUP_DATE}.tar.gz \
  /home/tom/traefik \
  /home/tom/wg-easy \
  /home/tom/vaultwarden \
  /home/tom/homeassistant \
  /home/tom/portainer \
  ~/homelab/.env

# Restart the stack
docker compose start
```

### Automated nightly backup (cron)

```bash
mkdir -p ~/backups
crontab -e
# Add this line to run at 02:00 every night:
0 2 * * * tar -czf /home/tom/backups/homelab-$(date +\%Y-\%m-\%d).tar.gz /home/tom/traefik /home/tom/wg-easy /home/tom/vaultwarden /home/tom/homeassistant /home/tom/portainer 2>/dev/null
```

> Offsite the backup archives to a NAS, cloud bucket, or external drive for resilience.

### Restore

```bash
# Stop the stack
docker compose stop

# Extract the backup
sudo tar -xzf ~/backups/homelab-2026-01-15.tar.gz -C /

# Restart
docker compose start
```

## Troubleshooting

### Container won't start

```bash
docker compose logs <service-name>
# e.g.:
docker compose logs traefik
docker compose logs homeassistant
```

### Traefik shows "certificate not trusted"

- Verify the cert files exist and are readable:
  ```bash
  ls -la /home/tom/traefik/certs/
  openssl x509 -in /home/tom/traefik/certs/local-cert.pem -noout -text | grep -A2 "Subject Alternative"
  ```
- Confirm the CA is imported on the device you're browsing from (see Step 3.4).

### Home Assistant shows "400 Bad Request" or logs "request from reverse proxy is blocked"

```bash
nano /home/tom/homeassistant/configuration.yaml
```
Ensure the `http.trusted_proxies` block contains `172.20.0.0/24` and restart:
```bash
docker compose restart homeassistant
```

### WireGuard VPN clients can't connect

1. Confirm UDP 51820 is open in UFW: `sudo ufw status`
2. Confirm port forwarding is set up on your router (UDP 51820 → NUC IP).
3. Check `WG_HOST` in `.env` matches your current public IP.

### Vaultwarden browser extension shows "invalid certificate"

The Bitwarden extension requires a **trusted** certificate. Make sure you have:
1. Imported the mkcert CA on the device running the extension.
2. Restarted the browser after importing.

### Disk space

```bash
# Show Docker disk usage
docker system df

# Clean up stopped containers, dangling images, unused networks
docker system prune -f

# Also remove unused volumes (careful — this deletes data not in a named volume)
docker system prune --volumes -f
```

### View live Traefik logs

```bash
tail -f /home/tom/traefik/logs/traefik.log
tail -f /home/tom/traefik/logs/access.log
```
